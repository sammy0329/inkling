# 문서 지도 (docs map)

이 폴더는 프로젝트의 "왜/무엇/어떻게 일하는가"를 담는다. 코드가 *무엇을 하는지*는 코드가 말하고, 문서는 코드가 말하지 못하는 **의도·결정·계약**을 담는다.

## 계층 (변경 주기로 나눈다)

정합성이 무너지는 가장 흔한 이유는 "안 바뀌는 것"과 "자주 바뀌는 것"을 한 파일에 섞기 때문이다. 그래서 변경 주기로 분리한다.

| 문서 | 담는 것 | 바뀌는 주기 |
|------|---------|-------------|
| `../CLAUDE.md` | 헌법: 스택·워크플로·로드맵 | 거의 안 바뀜 |
| `WORKFLOW.md` | 우리가 일하는 방식(운영 루프·검증 관문) | 드물게 |
| `decisions/` | ADR — "왜 이렇게 정했나" | append-only (안 지움) |
| `phases/` | 페이즈별 intent + 회고 | 페이즈당 1회 |
| `glossary.md` | 용어·상태값 단일 정의 | 도메인 바뀔 때 |
| `backend/README.md`, `frontend/README.md` | 실행법(how to run) | 실행 방식 바뀔 때 |

## 정합성 규칙 (딱 하나)

> **사실은 한 곳에만 적는다(single source of truth). 나머지는 링크한다.**

- 데이터 모델의 진실원은 `backend/src/db/schema.ts`다. 문서는 그걸 *설명*하되 필드를 복붙하지 않는다.
- 상태값(`draft`/`published`) 같은 도메인 용어의 진실원은 `glossary.md`다.
- 결정의 "왜"는 `decisions/`에만 있다. CLAUDE.md는 결론만, 상세 근거는 ADR로 링크.

## 읽는 순서 (새 세션/새 사람)

1. `../CLAUDE.md` — 이 프로젝트가 뭔지
2. `WORKFLOW.md` — 우리가 어떻게 일하는지
3. `phases/phase-N.md` — 지금 무슨 페이즈를 하는지
